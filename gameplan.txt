Concept

Use the autonomous Codex pipeline to generate a 3D web game from a natural-language prompt. Record the raw generated output, where it breaks, and how engine knowledge fixes each problem. The deliverable is a polished playable game, a clean repo, and a 5-7 minute YouTube video documenting the journey.

Game Target: 3D Dungeon Crawler

A first-person dungeon crawler in a single HTML file using Three.js. The player moves through procedurally placed rooms, collects items, avoids or fights enemies, and uses a minimap and health bar. This target is useful because it touches the hard parts of 3D web game generation: camera control, collision detection, lighting, level layout, enemy AI, UI overlay, and game state management. It is simple enough to build in a week and complex enough that an LLM will make real mistakes worth fixing on camera.

Phase 1: Spec and Generate (Days 1-2)

Feed the Codex pipeline a detailed game brief. Something like: "Build a 3D first-person dungeon crawler in a single HTML file using Three.js. The player explores connected rooms, collects keys to open doors, avoids patrolling enemies, has a health system, and a 2D minimap overlay. Use pointer lock controls, basic lighting with torches, and simple box and plane geometry for walls and floors."

Run `./scripts/generate-game.sh "<game idea>"` to create the spec, then run `./scripts/codex-coding-time.sh` to build the first artifact. Document everything: screenshot the raw output, note what works, and note what fails. Expect a playable but rough prototype with predictable problems.

Phase 2: Identify Failures (Day 3)

Play the generated game and catalog every issue. Common LLM 3D game failures to watch for:

- Camera clipping through walls because there is no collision offset on the camera
- Collision detection using naive distance checks instead of raycasts or solid layout data
- Enemies that walk through walls or get stuck
- Lighting that is either flat or far too dark
- The game loop mixing update logic with render logic
- No delta time, so the game runs at different speeds on different machines
- A minimap that does not match the actual level layout
- Memory leaks from creating new objects every frame instead of reusing them

List each failure clearly. This becomes the backbone of the repair plan and the video.

Phase 3: Fix with Engine Knowledge (Days 4-6)

Fix each problem one at a time. For each fix, leave a brief code comment or status note explaining why the pattern is better. These are the moments that make the video:

- "The AI put the player controller inside the render function, so I extracted it into its own update loop."
- "Collision detection was checking every mesh every frame, so I added a simple spatial partition."
- "The camera had no collision, so I added a short forward test and stop the player before wall penetration."
- "Enemy pathfinding was just moveToward(player), so I added waypoints or layout-aware steering."
- "There was no delta time, so the game ran twice as fast on a 120 Hz monitor."

Each fix is a 30-60 second segment in the video with clear before and after.

Phase 4: Polish (Day 7)

Make the game feel good to play. Smooth mouse look, responsive WASD, clear pickup feedback, readable lighting, and optional ambient sound. Host it on GitHub Pages so anyone can click a link and play it in a browser.

Clean up the repo. The final code should be readable and well-structured enough to show the difference between raw generated output and the repaired version.

Phase 5: Video (Days 8-9)

Suggested structure:

0:00-0:30 - Hook. Show the finished game. "I used AI to generate a 3D dungeon crawler in the browser. Here is what happened."
0:30-1:30 - The prompt and pipeline. Show the spec, the generated output, and the first browser run.
1:30-2:00 - "It works... mostly." Play the broken version and show the failures naturally.
2:00-5:00 - The fixes. One by one. Show code, then before and after in the game. Keep each segment short and concrete.
5:00-6:00 - The finished game side by side with the original AI output.
6:00-6:30 - Closing thought. The model gets most of the way there, but engine knowledge closes the gap.

No fancy editing is required. OBS, a mic, and clear narration are enough.

Push the repo, publish the video, and host the game on GitHub Pages.
